# ADR-0000 工程宪章

## 状态

已采纳。

## 背景

本仓库从一个几乎空的 Go 模块起步，必须在引入任何游戏逻辑、网络或集群代码之前，建立可审计的工程基线。基线须防止配置漂移、保持生成物确定性，并使开发机与 CI 上的仓库验证可信。

## 决策

本宪章授权建立本仓库的**初始工程治理基线**。

### 授权的根设计

1. **命令语义二分**：可变命令允许改动工作区；验证命令必须只读。
2. **单一事实源（SSOT）**：`.build/config.yaml` 是唯一由人编写的治理源；下游配置由脚本派生并纳入版本控制。
3. **宪章背书的初始规则**：初始规则集可引用本 ADR；后续每条硬约束须有独立 ADR。
4. **Proto 兼容性按基线存在性启用**：仅在配置的基线引用可解析时，破坏性检查才生效。
5. **可审计护栏**：验证不得依赖「干净」的 git 状态；CI 不得在检查前预先修复仓库；负例不得污染主模块；工具版本由 SSOT 约束。

### 根原则

1. 只读验证必须基于内容，而非 git 状态。
2. 派生文件为提升本地体验而提交，但禁止手工编辑。
3. 每条可执行约束须有 enforcer 与负例样本。
4. Bootstrap 假定主机已具备 `go` 与 `python3`（POSIX 级默认）；安装 `yq` 是**唯一**的零阶引导例外。
5. Hooks、CI 与手工验证须执行相同的验证目标集合。
6. 阶段演进通过对 `.build/config.yaml` 的显式修改完成。

### 初始规则授权

以下初始规则被授权引用本宪章，而无需各自独立 ADR：

- `project-overview`
- `go-standards`
- `architecture-boundaries`
- `mahjong-rule`
- `proto-standards`
- `testing`
- `commit`

每条 **constraint** 规则仍须声明 `enforcer` 与 `negative_test`。

### 零阶引导例外

在仓库尚无法完全依赖 `.build/config.yaml` 之前，`make bootstrap` 可按 `scripts/install-tools.sh` 内嵌的固定版本安装 `yq`。这是**唯一**允许出现在 SSOT 之外的硬编码工具版本。主机上的 `python3`（仅用标准库）可作为验证脚本与 shell、`yq` 的搭配依赖。

## 后果

- 治理变更通过单一人类可编辑文件流转。
- CI 能可靠发现过期的生成物与模块元数据。
- 主模块免受故意无效的负例污染。
- 未来约束须通过显式架构决策引入，而非挂靠宪章。
