# ADR-0028 跨地域多活

## 状态

草案。本 ADR 只评估拓扑，不做最终选型。

## 背景

ADR-0008 与 ADR-0011 已确定控制面、数据面与房间亲和路由，但默认假设单地域。
当业务需要跨地域容灾或低延迟接入时，需要先明确哪些状态可以跨区复制，哪些状态必须保持单主。

## 决策

### 1. 拓扑原则

- `gate` 可以多地域部署，优先靠就近接入降低握手延迟。
- `room` 以房间为最小亲和单元，同一房间同一时刻只能归属一个地域。
- PostgreSQL 事件日志以单主写入为起点，跨地域只读副本不承接房间事件写入。
- Redis 会话与幂等数据先按地域隔离，跨地域恢复需经过显式迁移流程。

### 2. 控制面候选

| 方案 | 优点 | 风险 |
| --- | --- | --- |
| 单地域 etcd + 跨区 gate | 实现简单 | 控制面地域故障会影响全局调度 |
| 多地域 etcd 独立集群 | 故障边界清晰 | 需要跨区房间迁移协议 |
| 全球一致控制面 | 语义统一 | 延迟、运维复杂度与脑裂风险较高 |

### 3. 故障切换

跨地域故障切换必须先冻结目标房间写入，再迁移路由与快照元数据。
未冻结前不得让两个地域同时接受同一房间动作。

## 后果

- 本 ADR 不修改 `internal/cluster` 代码。
- 后续若选择多地域 etcd 独立集群，需新增房间迁移 ADR。
- 跨地域能力不进入 Phase 6 默认交付范围，只作为 Phase 6+ 运维深化议题。
- Phase 6 收尾仅完成拓扑评估归档，不实施跨地域多活；后续立项须从本文候选方案中重新确认控制面与房间迁移边界。
