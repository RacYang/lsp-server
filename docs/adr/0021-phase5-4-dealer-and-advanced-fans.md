# ADR-0021 Phase 5.4 庄家与高阶番种

## 状态

已采纳。

## 背景

ADR-0020 已完成不依赖首巡语义的血战规则深化，并明确天胡、地胡需要庄家与开局窗口。当前 `rules.ScoreContext` 只描述胡牌来源、杠记录与海底等场况，无法判断胡牌方是否庄家，也无法把四杠类番种限定到胜家。

## 决策

### 1. 计分上下文

`internal/mahjong/rules.ScoreContext` 增加 `HuSeat`、`DealerSeat`、`IsOpeningDraw` 与 `IsDealerFirstDiscard`。房间引擎在调用 `ScoreFans` 时填充这些字段，规则包只消费结构化上下文，不反向依赖 room、handler 或 metrics。

### 2. 开局窗口

首局庄家为 seat 0，与当前 `StartGameNotify.DealerSeat` 保持一致。天胡采用保守口径：庄家首摸后立即自摸胡。地胡采用保守口径：闲家胡庄家第一张弃牌；闲家首摸自摸不计地胡。

开局窗口按当前胡牌动作的发生时刻判定，动作结算后再关闭，避免庄家第一张弃牌点炮时提前关闭地胡窗口。

### 3. 高阶番种

新增天胡、地胡、龙七对与十八罗汉。

- 天胡、地胡替代平胡、对对胡、将对、七对等基础牌型番；清一色、杠上炮、海底等场况番可继续叠加。
- 龙七对为七对且至少一组四张同牌；触发时替代七对，不再额外叠加一根。
- 十八罗汉按胡家自己的四笔杠记录判定；触发时替代暗杠与双暗杠，不再额外叠加这些杠番。

## 后果

- `round_json` schema 升级到 v3，新增庄家与开局窗口字段。旧 schema 缺少这些字段时按开局窗口已关闭恢复，避免错误补发天胡或地胡。
- 新番种必须在规则层、YAML 夹具与房间结算通知中保持同一中文标签。
- 未来若引入轮庄或第二套规则集，应先扩展庄家来源与规则选项，不在传输层硬编码番种口径。
